Identity

users

The one identity row per person. Everything else hangs off of user_id.

Fields

user_iduuid PK
created_attimestamp
display_nametext

Why

We need a stable, anonymizable identifier that survives renames and device changes. We don't store email or login creds here — auth lives in a separate identity provider so this table stays HIPAA-aligned.

Used by

every screen
Identity · cohort foundation

demographic_profile

The fields captured on the Demographics onboarding screen. This is the single most important table in the app — every cohort comparison starts here.

Fields

user_iduuid PK/FK1:1 with users
ageint→ cohort age window
sexenumF/M/NB/PNTS
race_ethnicitytext[]multi-select; optional
location_citytext
location_statechar(2)
height_inchesint→ BMI input
starting_weight_lbint→ BMI + weight Δ baseline

Why

Age + weight + height → matched-cohort age and BMI windows (the client's core ask: "how do my results compare to people like me?"). Sex → cohort sex filter and GLP-1 response differs by sex. Race/ethnicity → optional demographic lens for insights (never used as a treatment gate). Location → regional pricing variance on cost comparisons.

Used by

/demographics /profile CohortBadge /comparison /insights /adherence

User state · history

user_regimens

One row per regimen the user has ever been on. The row with ended_at IS NULL is the current one. Past rows preserve the drug timeline visible on Profile and drive historical side-effect / cost comparisons.

Fields

iduuid PK
user_iduuid FK
drug_idint FK
dosetexte.g., "5 mg"
frequencyenumweekly / daily / twice_weekly / other
started_atdate
ended_atdate?null = current
switch_reasonstext[]?set on end
decision_makerenum?self / doctor

Why

The medication timeline is a product feature (client: "what did I try before, why did I switch?"). By having event logs FK to regimen_id, a dose/side-effect/cost event stays tied to the exact drug + dose in use at the time — even after the user switches.

Used by

/dashboard regimen card /profile history /switch-medication all log screens

Worked example

How a cohort query uses these tables

To compute the "people like you" figures on Comparison and Insights, the backend joins a user's cohort_definitions row against every other user's demographic_profile, then aggregates that peer set's dose_logs, side_effect_logs, and cost_logs for the current category. 

-- For user U, find matched peers + their median 12-week weight change on Mounjaro
WITH my_cohort AS (
  SELECT dp.user_id
  FROM demographic_profile dp
  JOIN cohort_definitions cd ON cd.user_id = :me
  WHERE dp.age BETWEEN lower(cd.age_range) AND upper(cd.age_range)
    AND dp.sex = ANY(cd.sex_filter)
    AND bmi(dp.height_inches, dp.starting_weight_lb) <@ cd.bmi_range
)
SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY weight_change_pct)
FROM weight_change_12wk_mv
WHERE user_id IN (SELECT user_id FROM my_cohort)
  AND drug_id = :mounjaro_id;

In practice, heavy cohort aggregates (12-week weight change, side-effect incidence by drug, cost medians) live in materialized views refreshed nightly so the app itself only reads cheap summaries. The tables above are the raw sources of truth those views compute from.

Rules of thumb

QuestionRule
Does this field belong in Supabase or PreVeil?Structured data (enums, numerics, timestamps, FKs) → Supabase with RLS. Free-text clinical notes or whole-dossier exports → PreVeil.
Is this PHI?If it's a 1:1 fact about the patient's body or treatment (age, weight, drug, dose, adherence, side effect), yes. If it's a shared catalog (drug brand, category), no.
Should this be shown cross-user?Only through aggregate materialized views that contain no user_id. RLS + `SECURITY DEFINER` functions enforce this.
What about deletions?ON DELETE CASCADE from users removes every dependent row. PreVeil-stored exports need a separate deletion policy — PreVeil's admin console handles that.
