Why each table exists

Identity

users

The one identity row per person. Everything else hangs off of user_id.

Fields

user_id
created_at
display_nameoptional greeting-only

Why

We need a stable, anonymizable identifier that survives renames and device changes. No legal first/last name is stored — only an optional display name for greetings. Email and auth credentials live in a separate identity provider so this table stays HIPAA-aligned under the minimum-necessary principle.

Used by

every screen
Identity · cohort foundation

demographic_profile

The fields captured on the Demographics onboarding screen. This is the single most important table in the app — every cohort comparison starts here.

Fields

user_id1:1 with users
age→ cohort age window
sexF/M/NB/PNTS
race_ethnicitymulti-select; optional
location_city
location_state
height_inches→ BMI input
starting_weight_lb→ BMI + weight Δ baseline
comorbiditiesT2D / PCOS / HTN / CVD / GI / Pancreatitis / Thyroid / None

Why

Age + weight + height → matched-cohort age and BMI windows (the client's core ask: "how do my results compare to people like me?"). Sex → cohort sex filter and GLP-1 response differs by sex. Race/ethnicity → optional demographic lens for insights (never used as a treatment gate). Location → regional pricing variance on cost comparisons.

Used by

/demographics /profile CohortBadge /comparison /insights /adherence

Event · outcome

weight_entries

One row per weigh-in. The time series that powers the Dashboard and Insights weight-trend charts, the "You vs. cohort" weight-change row on Comparison, and the baseline-to-current Δ shown throughout the app.

Fields

id
user_id
weight_lb60–700 range check
recorded_atincludes time-of-weigh-in
fastedcaptured on Log weight
notes
sourcemanual / apple_health / google_fit / smart_scale

Why

Weight Δ is the single most asked-about metric in the app — the cohort pitch, the Comparison outcome row, and the Dashboard/Insights trend charts all depend on this time series. Keeping source on each row lets us distinguish a manually-entered Monday weigh-in from a Withings auto-sync so we can apply different confidence rules when computing trends. A unique(user_id, recorded_at) index prevents duplicate rows from simultaneous writes (e.g., manual entry racing an Apple Health HealthKit push).

Used by

/log-weight /dashboard weight trend /insights weight trend /comparison weight change clinician PDF export
Safety · screening

user_health_flags

Boxed-warning screening answers captured on the Medication onboarding safety check. Used to surface red-flag warnings and to exclude users from certain cohort-match comparisons where appropriate.

Fields

user_id
pregnancy_statusyes / no / prefer_not_to_say
mtc_men2_historyyes / no / prefer_not_to_say
updated_at

Why

GLP-1s carry a boxed warning for medullary thyroid carcinoma / MEN 2 history and are not recommended during pregnancy. Storing these as structured flags (not free text) lets the app show a red-flag callout on Medication onboarding and gate certain cohort comparisons (e.g., a pregnancy-flagged user shouldn't see a "people like you lost X%" claim that doesn't apply). Separate from demographic_profile so we can RLS-protect these columns more tightly.

Worked example

How a cohort query uses these tables

To compute the "people like you" figures on Comparison and Insights, the backend joins a user's cohort_definitions row against every other user's demographic_profile, then aggregates that peer set's dose_logs, side_effect_logs, and cost_logs for the current category. 

-- For user U, find matched peers + their median 12-week weight change on Mounjaro
WITH my_cohort AS (
  SELECT dp.user_id
  FROM demographic_profile dp
  JOIN cohort_definitions cd ON cd.user_id = :me
  WHERE dp.age BETWEEN lower(cd.age_range) AND upper(cd.age_range)
    AND dp.sex = ANY(cd.sex_filter)
    AND bmi(dp.height_inches, dp.starting_weight_lb) <@ cd.bmi_range
)
SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY weight_change_pct)
FROM weight_change_12wk_mv
WHERE user_id IN (SELECT user_id FROM my_cohort)
  AND drug_id = :mounjaro_id;

In practice, heavy cohort aggregates (12-week weight change, side-effect incidence by drug, cost medians) live in materialized views refreshed nightly so the app itself only reads cheap summaries. The tables above are the raw sources of truth those views compute from.

Rules of thumb

QuestionRule
Does this field belong in Supabase or PreVeil?Structured data (enums, numerics, timestamps, FKs) → Supabase with RLS. Free-text clinical notes or whole-dossier exports → PreVeil.
Is this PHI?If it's a 1:1 fact about the patient's body or treatment (age, weight, drug, dose, adherence, side effect), yes. If it's a shared catalog (drug brand, category), no.
Should this be shown cross-user?Only through aggregate materialized views that contain no user_id. RLS + `SECURITY DEFINER` functions enforce this.
What about deletions?ON DELETE CASCADE from users removes every dependent row. PreVeil-stored exports need a separate deletion policy — PreVeil's admin console handles that.
